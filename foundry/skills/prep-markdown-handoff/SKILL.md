# /foundry:prep-markdown-handoff

Final step of the `/foundry:prep-case` orchestrator pipeline. Reads the disk-persisted state
file produced by earlier sub-skills (`prep-npc`, `prep-scene`, `link-references`) and writes
every NPC, scene, and journal entry as an Obsidian-format `.md` file to a staging directory.
Then prints the exact manual instruction the MC needs to trigger the Obsidian Bridge import in
the Foundry UI.

**Closes:** R16 (markdown generation with YAML frontmatter + wikilinks) and R17 (explicit
manual-step handoff instruction).

---

## When to Use

- **Automatically**: invoked by `/foundry:prep-case` after `link-references` completes Pass 2.
- **Standalone**: re-run any time you want to regenerate or update the markdown staging files
  for a case that has already been prepped (e.g., after editing the state file to add a new NPC).

**Do NOT invoke before `link-references`** — the markdown output uses the resolved `@UUID[...]`
references in journals. Running before Pass 2 produces raw `@NPC[...]` / `@SCENE[...]`
placeholder tokens in the Obsidian files.

---

## Inputs

**Required argument:**

- `case_slug` — the slug key used for the `.foundry-prep/{case_slug}.json` state file.
  Example: `/foundry:prep-markdown-handoff case_0_session_2`

**Optional flags:**

- `--force` — overwrite every file in the staging directory. Default behaviour: **preserve**
  existing files and only add new ones. A re-run on a case that already has staging files will
  skip existing NPCs/scenes/journals unless `--force` is provided.
- `--vault-subdir <name>` — place all generated files inside a named subdirectory of the
  staging dir (e.g. `--vault-subdir case-0`). Useful if your Obsidian vault maps Foundry
  journal folders to subdirectories. Default: flat layout directly under the staging root.

---

## Execution Steps

### Step 1 — Load state file and verify preconditions

Read `.foundry-prep/{case_slug}.json`. If the file is missing, exit immediately:

```
Error: state file .foundry-prep/{case_slug}.json not found.
Run /foundry:prep-npc and /foundry:prep-scene first, then re-run this skill.
```

Check `state.uuid_placeholders_unresolved`. If `length > 0`, print a warning (do NOT exit):

```
Warning: link-references has {N} unresolved placeholders.
The generated markdown will contain raw @NPC[...] / @SCENE[...] tokens in affected journals.
Run /foundry:link-references first to resolve them, then re-run this skill.
```

The skill continues regardless — the user may want to ship the markdown anyway and resolve
later.

### Step 2 — Create the staging directory

Use Bash tool:

```javascript
const caseSlug = STATE.case_slug;

// Path traversal guard — case_slug must be a safe directory component
if (/[.\/\\]/.test(caseSlug) || caseSlug.startsWith('.')) {
  throw new Error(
    `case_slug "${caseSlug}" contains unsafe characters (., /, \\, or leading dot). Aborting.`
  );
}

const stagingDir = `.foundry-prep/obsidian-staging/${caseSlug}`;
// mkdir -p stagingDir
```

The guard rejects any `case_slug` containing `..`, `/`, `\`, or a leading `.`. The slugify
rule already prevents these by construction, but the skill double-checks defensively (T-04-24).

### Step 3 — Generate NPC markdown files

For each entry in `state.npcs[]`, write `.foundry-prep/obsidian-staging/{case_slug}/npc-{slug}.md`.

Skip files that already exist unless `--force` is set.

Template (substitute real values from state):

```markdown
---
title: {name} ({type})
aliases: [{name}]
tags: [npc, {type}, case-{case_slug}]
foundry-uuid: {foundry_actor_uuid}
foundry-type: Actor
---

# {name}

{short_description}

## Identity

- **Mythos:** {mythos}
- **Logos:** {logos}
- **Alias:** {alias}

## Themes

{for each theme: - **{theme.name}** — {theme.subtype} ({theme.mystery})}

## Power Tags

{for each power_tag: - {tag.name}{if tag.question: " — {tag.question}"}}

## Weakness Tags

{for each weakness_tag: - {tag.name}}

## GM Moves

{for each gm_move: - **{move.name}** ({move.subtype}): {move.description}}

## Relationships

{for each other NPC in state.npcs where other.slug != this.slug:}
{  - [[{other.name}]] — TBD}

---
*Generated by /foundry:prep-markdown-handoff for case {case_slug}. Foundry UUID: {foundry_actor_uuid}.*
```

Wikilinks use the NPC's human name as the link target. Obsidian Bridge resolves these against
other files in the staging directory.

### Step 4 — Generate scene markdown files

For each entry in `state.scenes[]`, write `scene-{slug}.md`.

Template:

```markdown
---
title: {name}
tags: [scene, case-{case_slug}]
foundry-uuid: {foundry_scene_uuid}
foundry-type: Scene
background: {background_path or "(none)"}
---

# {name}

## Background

Image: `{background_path}`

## Tokens

{for each token in scene.tokens: - [[{token.actor_name}]] at ({token.x}, {token.y})}

## Description

TBD — the GM may flesh out the scene description in Obsidian and re-import.

---
*Generated by /foundry:prep-markdown-handoff. Foundry UUID: {foundry_scene_uuid}.*
```

### Step 5 — Generate journal markdown files

For each entry in `state.journals[]`, write `journal-{slugified-journal-name}.md`.

The journal body was rewritten by `link-references` so Foundry `@UUID[Actor.xxx]{Name}`
references exist in the HTML. Convert them to Obsidian wikilinks:

**Conversion regex:** `@UUID\[(?:Actor|Scene|JournalEntry)\.[^\]]+\]\{([^}]+)\}` → `[[$1]]`

Raw unresolved placeholders (`@NPC[...]`, `@SCENE[...]`) are left as-is with an inline note:
`<!-- unresolved: @NPC[...] -->` appended on the same line.

Template:

```markdown
---
title: {name}
tags: [journal, case-{case_slug}]
foundry-uuid: {uuid}
foundry-type: JournalEntry
---

# {name}

{for each page in journal.pages:}
{  ## {page.name}}
{}
{  {page.content with @UUID → [[...]] conversion applied}}

---
*Generated by /foundry:prep-markdown-handoff. Foundry UUID: {uuid}.*
```

### Step 6 — Generate staging README.md (index + table of contents)

Write `.foundry-prep/obsidian-staging/{case_slug}/README.md` (always overwrite — this file
is regenerated on every run):

```markdown
# {case_name} — Obsidian Staging

Generated by `/foundry:prep-markdown-handoff`. Import via Obsidian Bridge in Foundry.

## NPCs

{bulleted wikilink list: - [[{npc.name}]] (`npc-{npc.slug}.md`)}

## Scenes

{bulleted wikilink list: - [[{scene.name}]] (`scene-{scene.slug}.md`)}

## Journals

{bulleted wikilink list: - [[{journal.name}]] (`journal-{slugified}.md`)}

## Unresolved References

{if uuid_placeholders_unresolved non-empty:}
{  | Source | Placeholder | Reason |}
{  |--------|-------------|--------|}
{  | {source} | {placeholder} | {reason} |}
{else: (none)}

## Missing Art

{if state.missing_art.npcs or state.missing_art.scenes or state.missing_art.items non-empty:}
{  | Entity | Expected path | Type |}
{  |--------|---------------|------|}
{  | {name} | {expected_path} | {type} |}
{else: (none)}
```

### Step 7 — Persist the handoff message to state file

Update `.foundry-prep/{case_slug}.json` — set `state.obsidian_handoff_message` to the
R17 handoff string (verbatim, with the real absolute path substituted). This lets downstream
skills or a re-run re-emit the message without regenerating the files.

### Step 8 — Print the R17 handoff message (REQUIRED)

Print the following to stdout. This is the R17 acceptance signal — the skill MUST emit it
every run, even if no files were written (e.g., all files existed and `--force` was not set):

```
Obsidian Bridge import required:
  1. Open Foundry VTT
  2. Select Obsidian Bridge in the module sidebar
  3. Import from folder → select: <absolute_path_to_staging_dir>
  4. Review the folder mapping and confirm

<N> markdown files ready for import.
```

Replace `<absolute_path_to_staging_dir>` with the resolved absolute path, e.g.
`/home/paul/dev/foundry-plugin/.foundry-prep/obsidian-staging/case_0_session_2/` (or the
Windows equivalent when running locally).

Replace `<N>` with the count of `.md` files in the staging directory after the run.

## Upsert / Re-Run Behaviour

| Mode | Behaviour |
|------|-----------|
| Default (no flags) | Preserve existing `.md` files. New NPCs/scenes/journals are written; existing files untouched. `README.md` always regenerated. |
| `--force` | Overwrite every file in the staging directory. Use after editing the state file or after `link-references` resolves new UUIDs. |

The state file's `obsidian_handoff_message` is updated on every run regardless of mode.

---

## Failure Modes

| Failure | Behaviour |
|---------|-----------|
| State file missing (`.foundry-prep/{case_slug}.json` not found) | Exit with clear error message. |
| `case_slug` contains `..`, `/`, `\`, or leading `.` | Reject immediately — path traversal guard (T-04-24). |
| File write permission denied | Surface the error, exit. Partial output is acceptable — re-run after fixing permissions. |
| `uuid_placeholders_unresolved.length > 0` | Print warning, continue. Raw placeholder tokens remain in affected journal files. |
| Obsidian Bridge not installed in Foundry | Irrelevant — skill only writes files. The user can install the Bridge module later or use the files directly in an Obsidian vault. |
| `state.missing_art` entries present | Record in `README.md` under "Missing Art" section; continue. Art absence does not block markdown generation. |

## Anti-Patterns

- **Do NOT call Obsidian Bridge via an API** — it has no API trigger (PITFALLS Gotcha 2). This
  skill only writes files and prints the handoff string. The import is always manual.
- **Do NOT write outside `.foundry-prep/obsidian-staging/{case_slug}/`** — path traversal guard
  is enforced in Step 2; never construct file paths from user input without validation.
- **Do NOT include raw Foundry document blobs in the frontmatter** — keep frontmatter minimal:
  `title`, `aliases`, `tags`, `foundry-uuid`, `foundry-type`. Full actor data lives in the body.
- **Do NOT skip the R17 handoff print (Step 8)** — it is the R17 acceptance signal and must
  appear in every run's output, even if the staging files already existed.
- **Do NOT make Foundry API calls** — this skill only writes local files. No `execute_js`
  calls, no relay requests. The deny-list (client.ts) is irrelevant here, but keeping the
  habit of local-only file writes is the design intent.

## Known Gaps

- **Frontmatter key uncertainty (A1):** `foundry-type` / `foundry-uuid` / `foundry-folder` are
  [ASSUMED] — not verified against `obsidian-bridge` module source because the module was not
  installed at research/execution time. Validated at Plan 08 UAT (round-trip import). If the
  keys differ, update Steps 3–5 and re-run with `--force`.
- **HTML → Markdown body conversion:** Journal page content may contain Foundry HTML (e.g.
  `<p>`, `<strong>`, `<em>`). This skill emits the HTML as-is inside the Obsidian markdown body.
  For best rendering in Obsidian, a future pass may convert the HTML to Markdown using a library
  such as `turndown`. Not required for Obsidian Bridge import — the Bridge accepts HTML in the
  body.
- **Flat staging layout only:** The `--vault-subdir` flag is documented but the exact Obsidian
  vault folder hierarchy expected by the Bridge is unknown until a round-trip import is tested.
  Default flat layout is the safest bet.
