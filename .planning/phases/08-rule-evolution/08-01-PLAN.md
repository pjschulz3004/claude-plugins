```typescript
// YAML schema — each rule file has this structure
export type RuleSource = "seeded" | "user_correction" | "self_generated";

export interface Rule {
  id: string;           // e.g. "email-001", "budget-001"
  text: string;         // The rule description
  confidence: number;   // 0.0-1.0 (per RULE-01, RULE-02)
  source: RuleSource;   // per RULE-03
  added: string;        // ISO 8601 date
  evaluations: number;  // How many times this rule has been evaluated
  accuracy: number;     // 0.0-1.0 — fraction of evaluations where rule was correct
}

export interface RuleFile {
  domain: string;       // "email" or "budget"
  version: number;      // Schema version (1)
  description: string;  // Human-readable purpose
  categories?: string[];  // Valid categories for this domain
  routing?: Record<string, string>[];  // Domain-specific routing info
  rules: Rule[];
}
```

RuleStore is a static utility class (no instance state). All methods are pure functions operating on RuleFile objects.

- `load(filePath: string): RuleFile` — reads file, parses YAML, validates structure
- `save(filePath: string, ruleFile: RuleFile): void` — serializes to YAML, writes to disk
- `addRule(ruleFile: RuleFile, rule: Omit<Rule, 'id'>): Rule` — generates next sequential id, pushes to rules array, returns the new rule
- `updateConfidence(ruleFile: RuleFile, ruleId: string, newConfidence: number): void` — finds rule by id, updates confidence. Throws if not found.
- `flaggedForReview(ruleFile: RuleFile): Rule[]` — returns rules where confidence < 0.8 (per RULE-04)
- `findByText(ruleFile: RuleFile, substring: string): Rule[]` — case-insensitive search

Use the `yaml` package (already in package.json) for parse/stringify. Use `node:fs` readFileSync/writeFileSync for file I/O.

Write tests first (TDD). Tests use a tmp directory with real YAML files — no mocking the filesystem. Test the round-trip (load->save->load produces identical RuleFile).
