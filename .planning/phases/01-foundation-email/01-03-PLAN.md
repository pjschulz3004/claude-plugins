**src/state/ledger.ts** -- TaskLedger:

Uses better-sqlite3 synchronous API. Constructor takes optional db path (defaults to "jarvis.db"; pass ":memory:" for tests).

Create the task_runs table on construction:
```sql
CREATE TABLE IF NOT EXISTS task_runs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  task_name TEXT NOT NULL,
  status TEXT NOT NULL CHECK(status IN ('success', 'failure', 'skipped')),
  started_at TEXT NOT NULL,
  duration_ms INTEGER NOT NULL,
  error TEXT,
  cost_usd REAL,
  input_tokens INTEGER,
  output_tokens INTEGER
);
CREATE INDEX IF NOT EXISTS idx_task_runs_name_started ON task_runs(task_name, started_at DESC);
```

Enable WAL mode on construction: `db.pragma('journal_mode = WAL')` (per Pitfall 8 from PITFALLS.md).

Methods:
- `record(entry: LedgerEntry): number` -- insert and return lastInsertRowid
- `getRecent(taskName: string, limit = 10): LedgerEntry[]` -- ORDER BY started_at DESC LIMIT
- `getConsecutiveFailures(taskName: string): number` -- count failures from most recent entry backward until a success or beginning
- `prune(olderThanDays = 30): number` -- DELETE WHERE started_at less than cutoff, return changes
- `close(): void` -- close the database

**src/dispatcher.ts** -- Dispatcher:

Uses `node:child_process` `execFile` (promisified via `node:util`). This is safe against shell injection since execFile does not spawn a shell -- arguments are passed directly to the executable.

Define the ClaudeResult interface:
```typescript
interface ClaudeResult {
  type: "result";
  subtype: "success" | "error_max_turns" | "error_api";
  result: string;
  session_id: string;
  total_cost_usd: number;
  duration_ms: number;
  usage: {
    input_tokens: number;
    output_tokens: number;
    cache_read_input_tokens: number;
  };
}
```

The `dispatch(prompt, opts)` method:
1. Build args: `["-p", prompt, "--output-format", "json", "--dangerously-skip-permissions"]`
2. Conditionally add: `--model`, `--plugin-dir` (repeated), `--max-turns`
3. Spawn via promisified execFile with timeout and 10MB maxBuffer
4. Parse stdout: try JSON.parse first. On failure, extract JSON with regex matching first `{` to last `}` (per Pitfall 4 from PITFALLS.md)
5. Validate subtype === "success", throw descriptive error otherwise
6. Return typed ClaudeResult

Per DAEMON-08: Do NOT set ANTHROPIC_API_KEY in env. The claude CLI uses the Max subscription automatically when no API key is present.

Add `addJitter(baseMs: number, maxJitterMs = 60_000): number` -- returns baseMs + random offset between 0 and maxJitterMs. Export for testing.

**src/scheduler.ts** -- Scheduler:

Reads heartbeat.yaml via the `yaml` package, creates `Cron` jobs from the `croner` package.

Constructor takes: `{ yamlPath: string, dispatcher: Dispatcher, ledger: TaskLedger, breakers: BreakerManager }`.

On each cron fire for a task:
1. Add jitter delay (setTimeout with addJitter(0, 60_000)) per DAEMON-09
2. Check `breakers.shouldAllow(task.service)` -- if false, record a "skipped" entry in ledger, return
3. Record start time
4. Call `dispatcher.dispatch(task.prompt, { model, pluginDirs, maxTurns, timeoutMs })`
5. On success: record success in ledger (include cost, tokens from ClaudeResult), call breakers.recordSuccess
6. On failure: record failure in ledger (include error message), call breakers.recordFailure

The `hours` field in heartbeat.yaml restricts when tasks fire. If current hour is outside the range, skip (record nothing -- croner can handle this via cron expression, but validate).

Methods:
- `start(): void` -- parse yaml, create Cron jobs
- `stop(): void` -- stop all Cron jobs
- `getTaskNames(): string[]` -- list registered task names

**src/main.ts** -- Daemon entry point:

```typescript
import { TaskLedger } from "./state/ledger.js";
import { BreakerManager } from "./state/breakers.js";
import { Dispatcher } from "./dispatcher.js";
import { Scheduler } from "./scheduler.js";
import { HealthServer } from "./health.js";
import { fileURLToPath } from "node:url";
import { dirname, join } from "node:path";

const __dirname = dirname(fileURLToPath(import.meta.url));

const ledger = new TaskLedger(join(__dirname, "..", "jarvis.db"));
const breakers = new BreakerManager();
const dispatcher = new Dispatcher();
const scheduler = new Scheduler({
  yamlPath: join(__dirname, "..", "heartbeat.yaml"),
  dispatcher,
  ledger,
  breakers,
});
const health = new HealthServer({ breakers, ledger });

async function start() {
  console.log("[jarvis] Starting daemon...");
  await health.start();
  scheduler.start();
  console.log("[jarvis] Daemon running. Health at :" + (process.env.JARVIS_HEALTH_PORT || "3333") + "/health");
}

async function shutdown(signal: string) {
  console.log("[jarvis] Received " + signal + ", shutting down...");
  scheduler.stop();
  await health.stop();
  ledger.close();
  console.log("[jarvis] Shutdown complete.");
  process.exit(0);
}

process.on("SIGTERM", () => shutdown("SIGTERM"));
process.on("SIGINT", () => shutdown("SIGINT"));

start().catch((err) => {
  console.error("[jarvis] Fatal:", err);
  process.exit(1);
});
```

The daemon uses `--env-file=.env` (Node 22 native) to load credentials at startup. No dotenv dependency.

IMPORTANT: The daemon does NOT include Telegram bot yet (that is Phase 4). This is the skeleton with scheduler + dispatcher + health.

Per DAEMON-07: Graceful shutdown on SIGTERM stops scheduler (cancels cron jobs), closes health server (stops accepting connections), and closes ledger database.
