    - Test 4: detectEmailCorrections() when email was triaged as NEWSLETTER (mark_read) but user flagged it → records correction
    - Test 5: detectBudgetCorrections() with no recent budget decisions returns empty array
    - Test 6: detectBudgetCorrections() when transaction category matches original → no correction
    - Test 7: detectBudgetCorrections() when transaction was auto-categorised as "Groceries" but user moved it to "Dining Out" → records correction
    - Test 8: Both detection functions call CorrectionStore.recordCorrection() for each found correction
  </behavior>
  <action>
  **Design:** Correction detection works by comparing Jarvis's logged decisions against the current state of the world. It does NOT need real IMAP/YNAB connections in tests — use dependency injection with mock backends.

  1. Create `corrections.test.ts` with 8 tests. Use mock objects for ImapFlowBackend and YnabBackend. Use real in-memory TaskLedger and CorrectionStore.

  2. Create `corrections.ts` with two exported functions:

  ```typescript
  export interface CorrectionDetectionDeps {
    ledger: TaskLedger;
    corrections: CorrectionStore;
    email?: { getMessageFlags(uid: string): Promise<{folder: string, flags: string[]}> };
    budget?: { getTransaction(id: string): Promise<{category_name: string}> };
  }

  export async function detectEmailCorrections(deps: CorrectionDetectionDeps): Promise<number>
  export async function detectBudgetCorrections(deps: CorrectionDetectionDeps): Promise<number>
  ```

  **detectEmailCorrections logic:**
  1. Query task_runs for recent email_triage entries with decision_summary (last 24h)
  2. Parse JSON decisions from decision_summary
  3. For each decision, check current email state via the email backend mock
  4. Map triage actions to expected states: TRASH → "Trash" folder, IMPORTANT → has \\Flagged, NEWSLETTER → has \\Seen, INVOICE → has \\Flagged + $Invoice keyword
  5. If current state differs from expected, call corrections.recordCorrection() with original_decision = the action Jarvis took, corrected_decision = what the current state implies Paul wanted
  6. Return count of corrections found

  **detectBudgetCorrections logic:**
  1. Query task_runs for recent budget entries with decision_summary (last 24h)
  2. Parse JSON decisions (similar format: [{transaction_id, original_category, ...}])
  3. For each, check current YNAB category via budget backend
  4. If category changed, record correction
  5. Return count

  **Important:** The email backend doesn't currently have a `getMessageFlags` method. The detection function should accept a simpler interface — a lookup function `(uid: string) => Promise<{folder: string, flags: string[]}>` — so it can be mocked easily. In production, this will be wired to an IMAP call. For now, the interface is what matters.

  Same for budget: accept `(transactionId: string) => Promise<{category_name: string}>` rather than the full YnabBackend.

  Run tests — write minimal implementation to make all 8 pass.
