The healing skill procedure:

## Step 1: Parse Error Context
Read the error context provided in the dispatch prompt. Extract: task_name, consecutive failure count, list of recent error messages (with timestamps), service name.

## Step 2: Categorise Error Pattern
Classify the error into one of these categories:
- **connectivity**: IMAP/CalDAV/CardDAV/YNAB connection refused, timeout, DNS resolution failure
- **auth**: 401/403, credential expired, token invalid, login failed
- **rate_limit**: 429, too many requests, rate exceeded
- **data**: parsing error, unexpected response format, missing field
- **resource**: out of memory, disk full, process killed
- **unknown**: none of the above patterns match

Use the error messages to determine category. Multiple errors may show the same pattern (connectivity) or a progression (auth then connectivity).

## Step 3: Diagnose by Category

For **connectivity**:
1. Attempt to use the relevant MCP tools to test the connection (e.g., list_unread for IMAP, list_events for CalDAV)
2. If tool works: transient issue, likely resolved. Report "Transient connectivity issue, service recovered."
3. If tool fails: service is down or network issue. Check if multiple services are affected (indicates network-level problem).

For **auth**:
1. Attempt a simple read-only operation via MCP tools
2. If auth error confirms: escalate to Paul -- credentials need manual renewal
3. Report which credential set is affected (MAILBOX, CALDAV, YNAB, etc.)

For **rate_limit**:
1. This is self-resolving. Report the rate limit pattern.
2. Suggest reducing heartbeat frequency for the affected task if it happens repeatedly.

For **data**:
1. Attempt the operation that triggered the error
2. If reproducible: report the exact error and the data shape that caused it
3. If not reproducible: likely a transient API response issue

For **resource**:
1. Escalate immediately -- Paul needs to check VPS disk/memory
2. Report the specific resource constraint

For **unknown**:
1. Report all error messages verbatim
2. Escalate to Paul with full context

## Step 4: Take Action or Escalate
- If diagnosed as transient (connectivity recovered, rate limit passed): report recovery, no action needed
- If diagnosed as fixable (none currently -- healing agent is read-only for safety): report what would fix it
- If diagnosed as requiring human intervention (auth, resource, unknown): send escalation notification with:
  - Task name and failure count
  - Diagnosis category
  - Specific error messages
  - Recommended action for Paul

## Step 5: Summary
Produce a brief healing report following `${CLAUDE_PLUGIN_ROOT}/references/jarvis-voice.md` tone. Include: task name, diagnosis, action taken or escalation sent.

Logic:
1. Check consecutive failures via `ledger.getConsecutiveFailures(taskName)`. If < 3, return early (guard).
2. Get recent entries via `ledger.getRecent(taskName, 5)` to collect error messages.
3. Build error context string: task name, failure count, each error message with timestamp.
4. Build a prompt for `claude -p`:
   - Include the error context

   - Tell the agent to read `${CLAUDE_PLUGIN_ROOT}/references/jarvis-voice.md` for tone

5. Dispatch via `dispatcher.dispatch(prompt, { model: "sonnet", maxTurns: 8, timeoutMs: 180_000, pluginDirs: [...] })`
6. On success: send notification with the healing result (non-urgent)
7. On dispatch error (the healing itself failed): send urgent notification "Healing agent failed for {taskName}: {error}" -- do NOT throw, just log and notify

Add a re-entry guard: maintain a `Set<string>` of task names currently being healed. Check at entry, add on start, delete in finally block. This prevents the scheduler from dispatching multiple healing runs for the same task while one is already in progress.
