  <behavior>
    - CalDAVConfig validates with serverUrl, username, password fields
    - CalendarEvent has id, summary, start (ISO), end (ISO), location?, description?, allDay boolean
    - CalendarTodo has id, summary, due? (ISO), completed boolean, description?, priority?
    - Zod schemas parse valid data and reject invalid data
    - TsdavCalendarBackend.listEvents(start, end) returns CalendarEvent[]
    - TsdavCalendarBackend.listTodos() returns CalendarTodo[] with completed=false
    - TsdavCalendarBackend.createEvent(summary, start, end, opts) returns void
    - TsdavCalendarBackend.completeTodo(id) returns void
    - Connection-per-operation: each method creates DAVClient, calls login(), operates, no persistent state
    - Transient errors (ECONNRESET, ETIMEDOUT, ECONNREFUSED) are retried with backoff (3 attempts)
    - Auth errors are NOT retried
  </behavior>
  <action>
    1. Create package.json for @jarvis/calendar with dependencies: @jarvis/shared, @modelcontextprotocol/sdk, tsdav, zod. Follow jarvis-email/package.json exactly.
    2. Create tsconfig.json with composite:true, referencing ../jarvis-shared.
    3. Create src/types.ts with:
       - CalDAVConfig interface: { serverUrl: string, username: string, password: string }
       - CalendarEvent interface: { id: string, summary: string, start: string, end: string, location?: string, description?: string, allDay: boolean }
       - CalendarTodo interface: { id: string, summary: string, due?: string, completed: boolean, description?: string, priority?: number }
       - Zod schemas: CalendarEventSchema, CalendarTodoSchema, ListEventsInputSchema (startDate, endDate as ISO strings), CreateEventInputSchema (summary, startDate, endDate, location?, description?), CompleteTodoInputSchema (id), ListTodosInputSchema (empty)
    4. Create src/backend.ts with:
       - CalendarBackend interface: listEvents(start, end), listTodos(), createEvent(summary, start, end, opts?), completeTodo(id)
       - TsdavCalendarBackend class implementing CalendarBackend
       - Private withConnection<T> method: creates new DAVClient({ serverUrl, credentials: { username, password }, authMethod: 'Basic', defaultAccountType: 'caldav' }), calls login(), runs operation, no cleanup needed (stateless HTTP)
       - Retry logic matching ImapFlowBackend: TRANSIENT_CODES set, isTransientError(), 3 attempts, exponential backoff with configurable retryDelayMs
       - listEvents: fetchCalendars(), then fetchCalendarObjects() with timeRange filter, parse VEVENT from iCalendar data, extract UID, SUMMARY, DTSTART, DTEND, LOCATION, DESCRIPTION. Handle all-day events (DATE vs DATE-TIME).
       - listTodos: fetchCalendars(), fetchCalendarObjects() with filters for VTODO, parse VTODO, filter where STATUS != COMPLETED and COMPLETED property is absent
       - createEvent: build VCALENDAR/VEVENT string with UID (crypto.randomUUID()), SUMMARY, DTSTART, DTEND, optional LOCATION/DESCRIPTION, call createCalendarObject()
       - completeTodo: fetch the specific todo object by URL/etag, update its VCALENDAR string setting STATUS:COMPLETED and COMPLETED:{now}, call updateCalendarObject()
       - Constructor accepts CalDAVConfig and optional retryDelayMs (default 1000, 0 in tests)
    5. Create src/index.ts re-exporting types and backend (matching jarvis-email/src/index.ts pattern).
    6. Write tests FIRST (TDD):
       - src/types.test.ts: Zod schema validation for events and todos (valid data passes, invalid rejected)
       - src/backend.test.ts: Mock tsdav's DAVClient. Test listEvents returns parsed events, listTodos filters completed, createEvent builds valid iCal, completeTodo updates status. Test retry on transient error. Test no retry on auth error.
    7. Use `vi.mock('tsdav')` to mock DAVClient in backend tests.

    Key learnings from jarvis-email to apply:
    - Use local textResult() helper in MCP server (not shared toolResult) for SDK type compatibility
    - Accept retryDelayMs in constructor for deterministic test timing
    - Auth errors (401, "Unauthorized") must NOT be retried
