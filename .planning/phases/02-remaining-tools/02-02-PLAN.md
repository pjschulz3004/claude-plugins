  <behavior>
    - CardDAVConfig validates with serverUrl, username, password
    - Contact has id, fullName, emails (string[]), phones (string[]), organization?, addresses (string[]), notes?
    - ContactSummary has id, fullName, primaryEmail?, organization? (for search results)
    - Zod schemas parse valid and reject invalid data
    - TsdavContactsBackend.searchContacts(query) returns ContactSummary[] matching name, email, or org (case-insensitive substring)
    - TsdavContactsBackend.getContact(id) returns full Contact with all vCard fields
    - TsdavContactsBackend.createContact(contact) creates a new vCard and returns void
    - TsdavContactsBackend.updateContact(id, fields) updates existing vCard fields and returns void
    - Connection-per-operation pattern with retry on transient errors
  </behavior>
  <action>
    1. Create package.json for @jarvis/contacts with dependencies: @jarvis/shared, @modelcontextprotocol/sdk, tsdav, zod. Same structure as jarvis-email/package.json.
    2. Create tsconfig.json with composite:true, referencing ../jarvis-shared.
    3. Create src/types.ts with:
       - CardDAVConfig interface: { serverUrl: string, username: string, password: string }
       - Contact interface: { id: string, fullName: string, emails: string[], phones: string[], organization?: string, addresses: string[], notes?: string }
       - ContactSummary interface: { id: string, fullName: string, primaryEmail?: string, organization?: string }
       - Zod schemas: SearchContactsInputSchema ({ query: z.string() }), GetContactInputSchema ({ id: z.string() }), CreateContactInputSchema ({ fullName, emails?, phones?, organization?, addresses?, notes? }), UpdateContactInputSchema ({ id, fullName?, emails?, phones?, organization?, addresses?, notes? })
    4. Create src/backend.ts with:
       - ContactsBackend interface: searchContacts(query), getContact(id), createContact(data), updateContact(id, fields)
       - TsdavContactsBackend class implementing ContactsBackend
       - withConnection<T>: creates DAVClient({ serverUrl, credentials: { username, password }, authMethod: 'Basic', defaultAccountType: 'carddav' }), calls login(), runs operation
       - Retry logic identical to calendar/email: TRANSIENT_CODES, 3 attempts, exponential backoff, retryDelayMs param
       - searchContacts: fetchAddressBooks(), fetchVCards(), parse vCard data, filter by query matching FN, EMAIL, or ORG (case-insensitive includes), return ContactSummary[]
       - getContact: fetchVCards() for specific contact by URL, parse full vCard (FN, N, EMAIL, TEL, ORG, ADR, NOTE), return Contact
       - createContact: build vCard 3.0 string from input data (BEGIN:VCARD, VERSION:3.0, FN, N, EMAIL, TEL, ORG, ADR, NOTE, UID, END:VCARD), call createVCard()
       - updateContact: fetch existing vCard, modify fields, call updateVCard() with new data and etag
       - vCard parsing: extract fields using regex or line-by-line parsing. Handle multi-value fields (multiple EMAIL lines, multiple TEL lines). Handle folded lines (continuation lines starting with space/tab).
    5. Create src/index.ts re-exporting types and backend.
    6. Write tests FIRST (TDD):
       - src/types.test.ts: Zod schema validation tests
       - src/backend.test.ts: Mock tsdav's DAVClient. Test searchContacts filters correctly, getContact returns full details, createContact builds valid vCard, updateContact preserves existing fields. Test retry on transient error, no retry on auth error.
    7. Use `vi.mock('tsdav')` to mock DAVClient in backend tests.
