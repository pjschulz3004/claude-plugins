**docs/cutover-runbook.md** -- step-by-step migration procedure with these sections:

1. **Prerequisites**: List what must be done before starting (all phases complete, .env populated, npm run build passes, npm test passes)

2. **Shadow Mode** (MIG-01, MIG-02):
   - Copy .env from Python jarvis, adjust vars to match TS names
   - Use a SECOND Telegram bot token (JARVIS_TELEGRAM_BOT_TOKEN) to avoid 409 Conflict with the Python bot
   - Start TS daemon: `sudo systemctl start jarvis-daemon`
   - Both services run simultaneously -- Python on port 8085, TS on port 3334
   - Monitor for 24-48 hours: compare health endpoints, check Telegram for duplicate/missing notifications
   - Compare task ledger outputs: `sqlite3 jarvis.db "SELECT task_name, status, started_at FROM task_runs ORDER BY started_at DESC LIMIT 20;"`
   - Shadow mode comparison is manual: review both services' outputs side-by-side

3. **Cutover** (MIG-03):
   - Stop Python service: `sudo systemctl stop jarvis`
   - Swap Telegram bot token: update TS .env to use the PRIMARY bot token
   - Restart TS daemon: `sudo systemctl restart jarvis-daemon`
   - Verify health: `curl http://localhost:3334/health`
   - Verify Telegram: send /status to the bot
   - Disable Python service: `sudo systemctl disable jarvis`

4. **Verification**:
   - Health endpoint returns green
   - All heartbeat tasks fire on next schedule
   - Telegram commands respond
   - Uptime Kuma monitoring updated to port 3334

5. **Rollback** (if issues found):
   - Stop TS daemon: `sudo systemctl stop jarvis-daemon`
   - Restore Python bot token in Python .env if changed
   - Start Python service: `sudo systemctl start jarvis`
   - Verify Python health: `curl http://localhost:8085/health`
   - Investigate TS issues before retrying
