    <purpose>

    Wrap GSD execute-phase with Forge decomposition discipline. When forge_discipline is enabled in .planning/config.json, this workflow runs before each phase to:
    1. Identify which decomposition layers (L1-5) are relevant to the current phase
    2. Enforce types-first task ordering within the phase plan
    3. Route tasks to the correct specialist agents by layer
    4. Run the 1-shot prompt test after completion to validate foundation quality

    **Entry conditions:**
    - forge_discipline flag must be true in .planning/config.json (checked at start — exits silently if false)
    - Receives PHASE_NUM and PHASE_NAME from the caller (forge-autonomous or user invocation)

    **Usable standalone or via forge-autonomous.md**

    </purpose>

    <required_reading>

    Read these before starting:
    - `.planning/config.json` — check forge_discipline flag
    - `.planning/ROADMAP.md` — get phase goal and requirements for PHASE_NUM
    - `forge.local.md` — get current decomposition state

    </required_reading>

    ## 1. Check forge_discipline Flag

    Read the config:
    [bash command to read forge_discipline]

    If false or missing: display "Forge discipline is disabled (forge_discipline=false in config.json). Exiting." and stop silently.

    If true: display "Forge Discipline active for Phase {PHASE_NUM}: {PHASE_NAME}" and continue.
    </step>

    <step name="identify_layers">
    ## 2. Identify Decomposition Layers

    Read the phase goal from ROADMAP for PHASE_NUM:
    [gsd-tools command to get phase details]

    Analyze the phase goal and requirements to determine which layers are involved:

    Layer identification rules:
    - **L1 (Types)**: Phase creates data models, interfaces, schemas, types — always include for any implementation phase
    - **L2 (Logic)**: Phase implements business logic, transformations, algorithms
    - **L3 (Edge)**: Phase builds API endpoints, database queries, external integrations, file I/O
    - **L4 (UI)**: Phase creates UI components, pages, layouts, interactive elements
    - **L5 (Integration)**: Phase wires end-to-end flows, adds E2E tests, validates user workflows

    Display the layer analysis:
    ```
    ## Decomposition Analysis: Phase {PHASE_NUM} — {PHASE_NAME}

    | Layer | Type | Relevant | Agent |
    |-------|------|----------|-------|
    | L1 | Data Model / Types | {yes/no} | forge-backend |
    | L2 | Pure Logic | {yes/no} | forge-backend |
    | L3 | Edge Logic | {yes/no} | forge-backend |
    | L4 | UI Components | {yes/no} | forge-frontend + forge-designer |
    | L5 | Integration | {yes/no} | forge-tester |

    Execution order: {list of relevant layers in order}
    ```

    Update forge.local.md with the layer plan:
    [python3 command to update decomposition.current_layer and write to forge.local.md]
    </step>

    <step name="enforce_ordering">
    ## 3. Enforce Types-First Ordering

    Before execution begins, verify the PLAN.md files for this phase follow layer order.

    Read the plan files for PHASE_NUM:
    [bash to find and read plan files in the phase directory]

    Check that:
    - Tasks touching types/interfaces come before tasks touching logic
    - Tasks touching logic come before tasks touching API/DB
    - UI tasks come after backend tasks
    - Integration/E2E tasks come last

    If ordering is wrong, display a warning and reorder recommendation. Do NOT block execution — log the concern and continue. The discipline is advisory on ordering within an already-planned phase.

    Display:
    ```
    Ordering check: {PASS/ADVISORY}
    {If ADVISORY: list specific ordering concerns}
    ```
    </step>

    <step name="spawn_agents">
    ## 4. Route to Specialist Agents

    For each relevant layer, spawn the appropriate specialist agent with the phase context.

    **Layer → Agent routing:**

    For L1, L2, L3 (any backend layer is relevant):
    ```
    Task(agent="forge-backend", prompt="Phase {PHASE_NUM} — {PHASE_NAME}: [Layer description and specific tasks from the PLAN.md for this layer]")
    ```
    Update forge.local.md: increment agent_xp.forge_backend

    For L4 (UI relevant):
    ```
    Task(agent="forge-frontend", prompt="Phase {PHASE_NUM} — {PHASE_NAME}: [UI component tasks from PLAN.md]")
    Task(agent="forge-designer", prompt="Phase {PHASE_NUM} — {PHASE_NAME}: [Design system validation for UI tasks]")
    ```
    Update forge.local.md: increment agent_xp.forge_frontend, agent_xp.forge_designer

    For L5 (Integration relevant):
    ```
    Task(agent="forge-tester", prompt="Phase {PHASE_NUM} — {PHASE_NAME}: [Integration test scenarios and E2E flows]")
    ```
    Update forge.local.md: increment agent_xp.forge_tester

    **If no specialist layers identified** (pure config/docs phase):
    Skip agent spawning. Display: "Phase {PHASE_NUM} is infrastructure — no specialist routing needed."

    After all agents complete, update forge.local.md:
    [python3 to append PHASE_NUM to decomposition.layers_complete]
    </step>

    <step name="one_shot_test">
    ## 5. 1-Shot Prompt Test (Between Phases)

    This gate validates that the foundation built so far supports clean feature addition.

    Ask the user via AskUserQuestion:
    - **header:** "1-Shot Foundation Test — Phase {PHASE_NUM} complete"
    - **question:** "Describe a simple feature you'd want to add to what we just built."
    - **options:** "I'll describe a feature" / "Skip 1-shot test for this phase"

    **On "I'll describe a feature":**
    Attempt to add the described feature in a single focused prompt. Assess:
    - Did adding it require fighting the architecture?
    - Did it fit naturally into the existing types?
    - Did existing tests still pass?

    Display result:
    ```
    1-Shot Test: {PASS / ADVISORY}
    Feature attempted: {description}
    Result: {clean addition / required rework}
    {If rework needed: specific areas to address}
    ```

    If ADVISORY (rework needed), ask via AskUserQuestion:
    - **question:** "1-shot test revealed friction. How to proceed?"
    - **options:** "Address foundation issues before next phase" / "Log as tech debt and continue"

    On "Address foundation issues": pause and present specific findings. Let user decide fixes before returning control to caller.
    On "Log as tech debt": add entry to forge.local.md notes section and continue.

    **On "Skip 1-shot test":**
    Log "1-shot test skipped for Phase {PHASE_NUM}" to forge.local.md notes and continue.

    Update forge.local.md: set health_checks.integration_tested: true (if test passed)
    </step>
