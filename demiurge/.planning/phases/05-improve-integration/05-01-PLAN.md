The workflow implements three phases:

PHASE 0 — SETUP
- Parse $ARGUMENTS for --duration (default: 30m) and --category (default: security,quality)
- Validate categories against allowed list: security, quality, simplification, testing, performance, design
- Reject visual/ui and feature_creep with explicit error: "Visual/UI and Feature Creep are not valid in GSD phase context. Use standalone /improve for those."
- Enable autopilot: touch .autopilot && echo "AUTOPILOT ON for $(pwd)"
- Record START_TIME: START_TIME=$(date +%s)
- Parse duration into BUDGET_SECONDS: 30m=1800, 1h=3600, 2h=7200
- Read GSD context: cat .planning/STATE.md and .planning/ROADMAP.md to understand which phase this improvement pass covers
- Check git status — if dirty working tree, STOP: "Working tree is dirty — commit or stash changes before running improve-phase."
- Detect project type (read package.json, Cargo.toml, pyproject.toml, go.mod) and identify test command
- Count source files to calibrate cycle depth

PHASE 1 — IMPROVEMENT CYCLES
Run cycles in the requested CATEGORIES order. Security always runs first if included, regardless of input order.

Before each cycle, run the HARD TIME GATE (this is mandatory, not optional):
  ELAPSED=$(($(date +%s) - START_TIME))
  REMAINING=$((BUDGET_SECONDS - ELAPSED))
  If REMAINING < 300: skip to Phase 2 immediately.

Per-cycle protocol (Analyse → Fix → Verify → Commit → Log):
1. Analyse: Spawn Task agent (Sonnet model) to scan codebase for category issues. Return top 3 findings with file paths and line numbers. Category lenses: sharp-edges for security, kaizen:analyse for quality, via-negativa for simplification, property-based-testing for testing, impeccable:optimize for performance, impeccable:clarify for design.
2. Fix: Apply top 3 findings. Never change test expectations, never delete tests, never introduce new dependencies without strong justification, keep changes atomic.
3. Verify: Run project test suite. If no tests: run compile check (tsc --noEmit / cargo check / python -m py_compile / go vet). On failure: revert the failing change, record in deferred findings.
4. Commit: git add -A && git commit -m "improve(<category>): <concise description>"
5. Log: Record cycle result in-memory (do NOT write to .improve-log.md).

Stall detection: if two consecutive cycles produce identical findings or no actionable improvements, stop cycling and proceed to Phase 2.

After all requested categories complete: if REMAINING > 300, start a second rotation (deeper pass using deferred findings as starting point). If second rotation stalls, proceed to Phase 2.

PHASE 2 — FINAL VERIFICATION AND OUTPUT
1. Run full test suite one final time.
2. Check git log for all commits made during this session.
3. Disable autopilot: rm -f .autopilot && echo "AUTOPILOT OFF"
4. Write VERIFICATION.md to project root. The file must begin with a YAML frontmatter block (delimited by triple-dashes) containing these fields:
   - phase: [current phase from STATE.md]
   - generated_by: forge-improve-phase
   - status: passed (if no HIGH-confidence deferred findings remain and tests pass) OR gaps_found (if any HIGH-confidence findings were deferred or tests failed)
   - duration_budget: [parsed duration, e.g. 30m]
   - categories_run: [comma-separated list]
   - cycles_completed: [count]
   - commits_made: [count]
   After the frontmatter, the document body includes:
   - "# Improve Phase Verification" heading
   - Summary table: Category | Findings | Fixed | Deferred
   - "## Fixed Issues" section: one bullet per commit (category, file, what changed and why)
   - "## Deferred Findings" section: ranked list of issues found but not addressed
   - "## Verification Result" section: status line and explanation
